# QualityPilot 使用说明

这份文档只解决一个问题：项目启动后到底应该怎么点、怎么看、怎么给面试官讲。

如果你想知道每个页面具体填什么测试数据，看这份更细的表：
[module_test_data_guide.md](module_test_data_guide.md)。

## 1. 正确启动

在项目根目录执行：

```powershell
.\.venv\Scripts\python.exe scripts\start_fullstack.py
```

打开：

```text
http://127.0.0.1:5173
```

如果端口被占用：

```powershell
.\.venv\Scripts\python.exe scripts\start_fullstack.py --api-port 8010 --web-port 5174
```

打开：

```text
http://127.0.0.1:5174
```

不要只启动 Vue。只启动 Vue 会导致页面能打开，但后端数据拿不到。

## 2. 这个项目到底是什么

QualityPilot 不是单纯的 AI 聊天页面，也不是只跑 pytest 的脚本。

它是一个测试开发平台原型，主线是：

```text
接口测试设计 -> AI 生成/裂变测试用例 -> pytest 自动化执行
-> JUnit / Allure 报告解析 -> RAG 检索上下文 -> AI 失败分析 / Bug 报告
```

你面试时可以把它讲成：

```text
我做的是一个基于 Vue + FastAPI + pytest + Allure + RAG + MCP 的智能自动化测试平台。
它把接口测试、用例管理、自动化执行、报告解析、知识库检索和 AI 分析放到一个工作台里。
```

## 3. 第一次演示按这个顺序

### 第一步：平台总览

打开首页，看 4 个数字：

- 测试用例
- API 接口
- 自动化场景
- 报告产物

如果这里不是数字，而是 `Failed to fetch`，说明后端没有连上，重新执行：

```powershell
.\.venv\Scripts\python.exe scripts\start_fullstack.py
```

### 第二步：API 测试

进入 `API 测试`。

你先做这几个动作：

1. 左侧选择登录接口。
2. 不改任何参数，直接点击发送请求。
3. 看响应体、状态码、耗时和 JSON 断言结果。
4. 点击 `AI 裂变用例`。
5. 看系统生成的正常、异常、边界、安全类用例。
6. 点击 `导出 cURL`。

这一页面试时重点讲：

```text
我把接口 method、path、headers、query params、body、环境变量、Mock、断言和关联 pytest 场景都结构化了。
这不是简单发请求，而是把接口测试资产平台化。
```

### 第三步：测试用例

进入 `测试用例`。

看这里的用例分类：

- API
- UI
- 功能
- 异常
- 安全
- 回归

这一页面试时重点讲：

```text
测试平台不只是执行，还要维护测试资产。
这里把接口、场景、pytest 目标和断言关联起来，方便后续执行和报告追踪。
```

### 第四步：自动化执行

进入 `自动化执行`。

选择：

```text
api_login
```

点击运行。

看：

- run_id
- passed / failed
- stdout / stderr
- JUnit XML 路径
- Allure results 路径

这一页面试时重点讲：

```text
前端触发后端执行 pytest，后端只允许执行预定义 scenario_id，避免页面传任意命令造成风险。
```

### 第五步：测试报告

进入 `测试报告`。

看：

- JUnit XML
- Allure results
- 通过数
- 失败数
- 产物路径

这一页面试时重点讲：

```text
自动化平台不能只跑测试，还要解析测试结果。
我用 JUnit XML 做程序解析，用 Allure 产物做人看的报告。
```

### 第六步：知识库管理

进入 `知识库管理`。

可以输入：

```text
login token 401
```

看召回片段：

- source_type
- source_id
- title
- score
- content

这一页面试时重点讲：

```text
AI 生成测试用例不能只靠模型猜，所以我设计了 RAG 知识库。
需求、接口文档、历史 Bug、测试报告、日志和规范都可以作为测试上下文。
```

### 第七步：AI 测试助手

进入 `AI 测试助手`。

选择模板：

- 智能用例生成
- 接口测试设计
- 失败原因分析
- Bug 报告生成

输入需求后生成结果。

这一页面试时重点讲：

```text
AI 助手不是普通聊天，而是围绕测试开发任务做了提示词模板和结构化输出。
它可以复用 RAG 检索结果，生成测试用例、失败分析和 Bug 报告。
```

## 4. 你应该先练哪几个功能

先练这 4 个：

1. `API 测试`：发送登录接口请求，看断言结果。
2. `API 测试`：点击 AI 裂变用例，看生成的异常和安全用例。
3. `自动化执行`：运行 `api_login`。
4. `测试报告`：看 JUnit / Allure 产物。

这 4 个能讲清楚，项目就已经像测试开发项目了。

RAG、MCP、AI 助手是加分项，不要一开始就先讲它们。

## 5. 面试官问“AI 驱动体现在哪里”怎么答

可以这样答：

```text
AI 不是替代自动化测试框架，而是辅助测试开发工作流。
它主要体现在三个地方：
第一，根据接口或需求生成测试用例；
第二，通过 RAG 检索需求、接口文档、历史 Bug 和测试报告作为上下文；
第三，对失败用例进行原因分析，并生成 Bug 报告草稿。
真正执行测试的部分仍然交给 pytest 和报告解析链路完成。
```

## 6. 当前项目还有什么不足

可以诚实说：

```text
当前版本已经能演示 API 测试、自动化执行、报告解析、RAG 检索和 AI 产物生成。
但还没有做到企业级测试平台，后续会继续补：
1. 用例和环境配置持久化；
2. Web 自动化测试工作台；
3. 性能测试场景；
4. 执行队列和任务状态轮询；
5. 更完整的 Allure HTML 报告展示。
```
