# AI 自动化测试平台面试问答稿

这份文档用于回答面试中常见的三个问题：

- 项目中体现了哪些测试设计和测试用例？
- 自动化测试做了什么，用了什么工具，流程和结果是什么？
- 性能测试怎么讲，用了什么工具，测哪些接口，结果是什么？

## 1. 项目中如何体现测试设计

可以这样回答：

> 这个项目不是只做一个 AI 页面，而是围绕测试开发工作流做了完整闭环。我在项目中体现了需求分析、测试点生成、测试设计评审、自动化执行、测试报告、执行历史和质量趋势。测试用例不是只写了一个登录，而是覆盖了 API、UI、RAG 检索、执行计划、报告解析、历史记录和 CI 回归这些平台核心功能。

项目里主要测试了这些页面和功能：

| 页面 / 模块 | 主要测试点 |
| --- | --- |
| 测试工作台 | 输入需求、选择测试维度、生成 Markdown 测试点、RAG 知识片段引用 |
| 测试设计评审 | 检查测试点是否缺少边界、异常、安全、回归等维度，识别空泛用例和不可执行断言 |
| 追踪矩阵 | 将需求项关联到测试点、自动化场景和最近执行结果，展示覆盖缺口，并可导出评审材料 |
| 测试设计评估 | 使用 Golden Test Set 评估需求覆盖率、维度覆盖率、引用覆盖率和空输出 |
| 自动化场景 | 展示内置 API 登录、API 文件上传、UI 登录冒烟场景 |
| 执行计划 | 将自然语言步骤解析成结构化 action，并选择 API 或 Browser adapter 执行 |
| 执行历史 | 保存每次执行的场景、状态、失败原因、报告路径和 artifact |
| 测试报告 | 解析 JUnit XML，发现 Allure results / Allure report 目录 |
| CI | GitHub Actions 自动运行核心回归并上传 JUnit、Allure results 和 Allure HTML artifact |

## 2. 测试用例设计覆盖了哪些方面

面试时可以按测试类型讲：

### 功能测试

重点验证平台核心链路是否可用：

- 输入需求后能生成结构化测试设计草稿。
- RAG 检索结果能展示来源、分数、片段内容和 source type。
- 执行计划能识别 `open`、`input`、`click`、`assert_text`、`POST /api/login` 等步骤。
- API adapter 能执行登录、上传、断言响应文本。
- Browser adapter 能 dry-run UI 步骤，真实 Playwright 环境可扩展执行。
- 报告中心能解析 JUnit XML 和发现 Allure 目录。

### 接口测试

项目内置了 API 登录和文件上传场景：

- 登录成功：账号密码正确，返回 token。
- 登录失败：密码错误，返回错误状态和错误信息。
- 文件上传成功：带文件名和二进制内容，返回 filename 和 size。
- 文件上传失败：缺少文件名或内容，返回错误。

### 界面测试

使用 Streamlit AppTest 做 Dashboard 页面 smoke test：

- 平台总览页面能正常渲染。
- 测试工作台页面能正常渲染。
- 自动化场景页面能正常渲染。
- 执行计划页面能正常渲染。
- 执行历史页面能正常渲染。
- 追踪矩阵页面能正常渲染。
- 测试报告、测试设计评估、测试设计评审页面能正常渲染。

### 兼容性测试

当前项目的兼容性重点是环境和输出格式：

- Python 3.10+ 兼容。
- 本地 Windows 能运行。
- GitHub Actions Ubuntu 环境能运行。
- JUnit XML 能被报告中心解析。
- Allure results 使用通用 JSON 附件格式，可通过本地脚本或 CI 生成静态 HTML 报告。

### 易用性测试

主要验证测试人员使用时是否容易理解：

- Dashboard 页面有清晰入口。
- 执行计划支持 dry-run，避免必须准备真实环境。
- 无历史记录、无报告、无知识片段时都有空状态提示。
- README 提供启动命令、演示顺序和核心回归命令。

### 安全性测试

主要覆盖：

- 不提交明文密钥。
- `.gitignore` 忽略 `data/`、`reports/`、`logs/`、`.env` 等本地敏感或生成文件。
- CI 默认不依赖任何 secret。
- API adapter 测试覆盖未认证、错误凭证、token 断言等场景。

### 弱网 / 异常测试

当前项目用确定性方式模拟异常：

- API adapter 处理网络连接失败。
- Browser adapter 在 Playwright 未安装时返回可读失败原因。
- 执行计划遇到不支持步骤时不会静默丢弃，而是记录 warning。
- 报告文件缺失时，测试报告页面会提示用户先生成报告。

### 性能测试

当前项目没有把性能压测作为主要实现，但可以这样讲：

> 第一版重点是 AI 测试设计和自动化执行闭环，所以性能测试没有做大规模压测平台。但我设计了可以扩展的方向：对 API 登录、文件上传、RAG 检索、测试点生成和报告解析这些核心接口或函数做响应时间、吞吐量和错误率统计。后续可以接入 Locust、JMeter 或 pytest-benchmark。

## 5. 自动化测试具体流程

可以这样讲：

1. 先在测试工作台或预设场景里准备测试目标。
2. 对 API 或 UI 场景生成自然语言步骤。
3. 执行计划服务把自然语言步骤解析成结构化 action。
4. 根据 action 类型选择 API adapter 或 Browser adapter。
5. adapter 执行或 dry-run 每一步，并记录 passed、failed、skipped、dry_run。
6. 执行结果写入 JUnit XML 和 Allure results。
7. 可选生成 Allure HTML，并在 CI 中作为 artifact 上传。
8. 可选保存执行历史。
9. 追踪矩阵把需求项、测试点、自动化场景和最近执行状态关联起来。
10. 测试报告页面读取报告。
11. 执行历史页面展示趋势、失败原因和 artifact。
12. GitHub Actions 在 push 后自动跑核心回归。

## 6. 自动化测试结果是什么

当前阶段本地完整回归结果：

```text
93 passed
```

可以强调：

- 所有核心服务都有单元测试。
- Dashboard 页面有 smoke test。
- API 和 UI demo 场景有自动化回归。
- CI 会在 GitHub 上自动跑测试。
- JUnit、Allure results 和 Allure HTML 报告会以 artifact 形式保存。

如果面试官问有没有通过自动化发现代码问题，可以这样说：

> 有。开发过程中自动化测试发现过报告状态映射、CLI 参数兼容、Dashboard 页面导入等问题。比如 Allure result 写入时状态常量没有正确导入，单元测试立即暴露了问题；修复后再跑完整回归，避免了报告链路上线后才发现失败。

如果面试官问如何证明测试覆盖需求，可以这样说：

> 我加了追踪矩阵页面，它会把需求拆成 REQ-001、REQ-002 这样的条目，再从测试设计 Markdown 中提取测试点，按关键词和维度做确定性匹配。同时它会匹配内置自动化场景，比如 API 登录、文件上传、UI 登录冒烟，并从执行历史里取最近一次执行状态。这样可以回答每条需求有没有测试点、有没有自动化、最近是 passed、failed、dry-run 还是没跑过。

如果面试官问如何把结果给团队评审，可以补充：

> 追踪矩阵支持给每条需求标记评审状态，比如已确认、需补测试设计、需补自动化或阻塞待处理，并且能导出 Markdown 和 CSV。Markdown 适合放到评审文档或 PR 描述里，CSV 适合做表格筛选和统计。

## 7. 性能测试怎么讲

如果面试官重点问性能测试，可以按下面结构回答：

> 当前项目没有把性能压测作为主线功能，因为面试项目的核心目标是 AI 测试设计、自动化执行和报告闭环。但如果要做性能测试，我会优先选择 RAG 检索接口、测试点生成服务、API adapter 执行接口和报告解析服务作为压测对象。

### 可以使用的工具

- JMeter：适合接口压测和生成聚合报告。
- Locust：适合用 Python 编写用户行为模型。
- pytest-benchmark：适合函数级性能基准，比如报告解析、规则评审、趋势聚合。
- GitHub Actions：适合跑轻量性能基线，不适合跑大规模压测。

### 会测哪些接口或函数

- RAG 检索：关注 P95 延迟、召回耗时、错误率。
- 测试设计生成：关注生成耗时、输出长度、失败率。
- API adapter：关注接口响应时间、错误处理耗时。
- 报告解析：关注大 JUnit XML 文件解析耗时。
- 执行历史趋势聚合：关注大量历史记录下的统计耗时。

### 性能测试流程

1. 确定压测目标和指标，比如 P95 < 500ms。
2. 准备固定测试数据，避免数据波动影响结果。
3. 使用 Locust 或 JMeter 设计并发用户模型。
4. 从低并发逐步加压，比如 10、50、100、200 并发。
5. 观察响应时间、吞吐量、错误率、CPU、内存。
6. 分析慢接口或慢函数。
7. 优化后重新压测，形成对比报告。

### 如果没有真实性能问题怎么说

可以这样回答：

> 目前没有在项目中制造真实性能瓶颈，所以不会夸大说发现了严重性能问题。但我会讲清楚性能测试设计：哪些接口需要测、怎么准备数据、怎么加压、看哪些指标、如何定位问题。如果后续扩展，我会先对 RAG 检索和报告解析做性能基线，因为这两个模块最容易随着数据量增长出现耗时问题。

可以直接这样说：

> 我这个项目里测试不是只写了几个 pytest，而是围绕 AI 自动化测试平台做了分层测试。功能上测了测试工作台、测试设计评审、追踪矩阵、执行计划、API/UI adapter、报告中心、执行历史和 CI；自动化上用了 pytest、Streamlit AppTest、JUnit XML、Allure results、Allure HTML artifact 和 GitHub Actions；用例设计覆盖功能、接口、界面、异常、安全、兼容性、易用性和弱网异常。性能测试当前没有做大规模压测，但我预留了 RAG 检索、测试设计生成、报告解析和历史趋势聚合这些压测对象，后续可以用 Locust、JMeter 或 pytest-benchmark 做性能基线。
